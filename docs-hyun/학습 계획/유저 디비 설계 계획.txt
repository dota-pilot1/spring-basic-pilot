[학습 계획: 유저 및 게시판 DB 설계]

이 계획은 데이터베이스를 단순히 만드는 것을 넘어, 어떻게 구조적으로 '설계'하고 '검증'하는지 배우는 과정입니다.

1단계: 요구사항 정의 및 엔티티 도출
- 질문: 우리 서비스에 어떤 정보가 필요한가?
  - 유저: 아이디, 비밀번호, 권한, 가입일
  - 게시판: 제목, 내용, 작성자, 조회수, 작성일
- 학습 포인트: 핵심 속성 추출 및 데이터 타입(String -> VARCHAR, Date -> TIMESTAMP 등) 매핑

2단계: 관계 설계 (ERD 기초)
- 개념: "유저 한 명은 여러 개의 게시글을 쓸 수 있다" (1:N 관계)
- 설계:
  - `boards` 테이블에 `author_id` 컬럼을 생성하여 `users` 테이블의 `id`를 참조(FK)하게 함.
- 학습 포인트: Primary Key(PK)와 Foreign Key(FK)의 개념 이해

3단계: 물리 설계 및 스키마 작성
- 도구: DBeaver에서 쿼리 직접 쳐보기 -> `schema.sql`에 기록
- 제약 조건 설정:
  - `NOT NULL`: 필수 입력값 설정
  - `UNIQUE`: 중복 아이디 방지
  - `DEFAULT`: 초기값(조회수 0 등) 설정
- 학습 포인트: 튼튼한 테이블을 만드는 제약 조건(Constraints)의 중요성

4단계: 데이터 정합성 검증 (DBeaver 실습)
- 실습: 
  - 존재하지 않는 유저 ID로 게시글을 써보기 (FK 에러 확인)
  - 탈퇴한 유저의 글 처리 시나리오 고민 (ON DELETE CASCADE 등)
- 학습 포인트: DB 차원에서 데이터가 꼬이지 않게 막아주는 기능 확인

5단계: 테스트 데이터(Seed Data) 준비
- 목적: 화면 개발 시 데이터가 없어서 생기는 불편함 해소
- 상세: 최소 2명 이상의 유저와 각 유저당 1~2개의 게시글 데이터 생성
- 학습 포인트: 초기 데이터 설정의 효율성
